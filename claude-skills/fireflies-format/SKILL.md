# fireflies-format

Convert raw Fireflies notes into a consistently structured Markdown file. Fireflies pastes come with no heading hierarchy — just bold-ish leading phrases, naked section titles, timestamps trailing each subsection, and a flat "Action items" block grouped only by name. This skill imposes a consistent shape on that.

## When to invoke

- User pastes Fireflies-style notes and asks to format / clean up / rearrange / restructure them.
- User opens a file under `meetings/**` containing raw Fireflies text and asks for proper markdown structure.
- User says things like "make this a proper markdown" or "fix this meeting note".

Do **not** invoke for: notes the user has already structured by hand, generic summarization tasks, or non-meeting markdown.

## What raw Fireflies output looks like

A typical paste has three regions, in this order:

1. **Top bullets** — 4–8 lines, each starting with a "Theme: explanation" pattern, separated by blank lines. No heading.
2. **`Notes`** — a flat list of section titles, each followed by:
   - a one-line description,
   - one or more subsection titles each ending in a `(MM:SS)` timestamp,
   - 2–4 bullet-like sentences under each subsection (often without `-` markers).
3. **`Action items`** — flat list. Person name on its own line, then their action items (each ending in a `(MM:SS)` timestamp), then the next person's name, etc.

There is usually no `#` heading anywhere, code fences are absent, and bullets are missing their `-`.

## Target structure

Produce a single markdown file with this skeleton:

```markdown
# <Meeting title> (<YYYY-MM-DD>)

## Key Takeaways

- **<Theme>** — <explanation>.
- ...

---

## Notes

### <Section 1 title>

<one-line section description>

#### <Subsection title> (MM:SS)

- <bullet>
- <bullet>

#### <Subsection title> (MM:SS)

- ...

---

### <Section 2 title>

...

---

## Action Items

### <Person name>

- [ ] <action> (MM:SS)
- [ ] <action> (MM:SS)

### <Person name>

- [ ] <action> (MM:SS)
```

## Rules

- **Title.** Derive from the filename stem if the user hasn't given one. Convert dashes to spaces and title-case it. Append the meeting date in `(YYYY-MM-DD)` form — pull from the filename if it ends in `DDMMYYYY` or similar, otherwise ask.
- **Key Takeaways.** Take the top bullets verbatim in meaning. Bold the leading theme (everything before the first `:` or `—`), then an em-dash, then the explanation as a sentence. Don't invent new takeaways.
- **Sections (H3) vs subsections (H4).** A Fireflies "section" is a title line with no timestamp followed by a one-line description. A "subsection" is a title with a `(MM:SS)` timestamp. Preserve that two-level structure exactly.
- **Bullets.** Every fact line under a subsection becomes a `-` bullet. Don't merge or reorder them.
- **Action items.** Convert to GitHub task list checkboxes (`- [ ] ...`). Keep the trailing `(MM:SS)` timestamp on each item. Group under `### <Person>` H3s.
- **Light copy-edits only.** Fix obvious typos, smart-quote artifacts, and capitalization of product names (HiBob, Confluence, Zoho, etc.). Do **not** rewrite sentences, change meaning, or summarize further.
- **Separators.** Use `---` between major regions (Key Takeaways / Notes / Action Items) and between top-level H3 sections inside Notes. Don't put `---` between H4 subsections.
- **Don't** add a "Generated by Fireflies" footer, attendee list you weren't given, or any content that wasn't in the source.

## Workflow

1. Read the target file (or the user's pasted text).
2. Identify the three regions (top bullets, `Notes`, `Action items`).
3. Rewrite into the target structure above, preserving all factual content and timestamps.
4. Overwrite the file with `Write` (or return the formatted text if the user pasted inline).
5. Briefly confirm what you changed in one sentence — don't re-summarize the meeting.

## Edge cases

- **Missing `Action items` block.** Skip that section entirely; don't fabricate one.
- **Subsection has no timestamp.** Keep it as H4 without a timestamp suffix.
- **Top bullets missing.** Skip "Key Takeaways"; start with `## Notes`.
- **Multiple meetings in one file.** Treat each as its own H1 block; ask the user if unsure.
- **Names in action items are ambiguous** (e.g., a single first name that could be either party). Keep them as written; don't guess full names.
